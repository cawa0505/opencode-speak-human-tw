## 提交前先想一件事

> 這是一個「新模式」，還是「現有模式的變體」？

「奠定基礎」和「立下根基」是同一個模式（誇大意義）的兩種說法。詞表只列代表項，不追求窮舉；skill 的執行原則是「模式優先、詞表兜底」，多數變體現有模式就吃得住。

只有滿足下面至少一條，才建議新增：

- 這個說法在多個獨立來源反覆被點名
- 它改變了誤殺邊界（現有規則會誤傷正常用法）
- 現有 36 種模式沒有一種能穩定吸收它

如果只是變體，更好的貢獻是：補一條評測用例、補一個範例句，或補 [references/patterns.md](references/patterns.md) 某個模式的識別信號。

## 提交新的 AI 味模式

PR 需包含四件事：

1. **模式名稱與問題描述**：為什麼這是 AI 味，不是正常寫作
2. **識別信號**：關鍵詞或句式結構，1–2 句講清楚
3. **繁體中文的 ❌/✅ 對比**：至少一組，範例句必須原創、合成，不指向真實人物或品牌
4. **誤殺邊界**：這個模式在什麼情況下是合理的？（例：排比底下有真實對比就放行）

同時在 [evals/benchmark.md](evals/benchmark.md) 補 SF/SNF 成對用例：一條「該改」，一條「不可誤殺」。

## 提交中國用語對照

[references/taiwan-localization.md](references/taiwan-localization.md) 的替換表歡迎補充，需說明：

- 台灣的對應說法（最好附一句實際用法）
- 是「一律替換」還是「按語境判斷」：像「優化」這種掛在具體指標上就合理的詞，要放進「按語境判斷」段落並寫清楚放行條件
- 有沒有專有名詞例外（平台名、產品名照原文）

## 提交評測用例

格式照 [evals/benchmark.md](evals/benchmark.md) 現有用例：

- **SF**：原文、問題點、預期方向
- **SNF**：原文、不能動的理由、預期行為

規範：

- 合成文本或已脫敏，不指向真實人物、品牌、課程；價格與名稱用虛構值
- SF 與 SNF 盡量成對提交
- 同步更新 benchmark.md 開頭的計數與覆蓋矩陣，CHANGELOG 記一筆

## 提交 bad case

遇到「用了 skill 改完還是像 AI」的真實文本，用 [bad case 模板](.github/ISSUE_TEMPLATE/bad-case.md)開 issue。這是對專案最有價值的貢獻，比 star 有用。

提交前請脫敏：不要貼未授權的私訊全文、個資、內部連結、金鑰。

## PR 規範

1. 一個 PR 只做一件事（加模式、加用語、加用例、修文件，擇一）
2. 更新 [CHANGELOG.md](CHANGELOG.md)：寫進 Unreleased 段落即可，版本號由維護者發版時定
3. 改了規則邏輯，同步更新 `SKILL.md` 與對應的 `references/` 檔案；兩邊不一致的 PR 不收
4. 改了 benchmark 數量或判分口徑，同步更新 [evals/run-eval.md](evals/run-eval.md)
5. 所有文件遵守本 skill 自己的規則：全形標點、破折號密度、不堆 emoji。改稿工具的文件自己有 AI 味，說不過去

## 維護者流程：新模型口癖巡檢

每當主流模型發布新版本（GPT、Claude、Gemini 等），它的中文口癖會變。維護者巡檢流程：

1. **收集**：用固定的探針任務（寫一則貼文、回一封客服信、寫一段課程文案）跑新模型，收 5–10 段輸出；同時看社群（Threads、FB 社團、PTT）有沒有多處獨立吐槽同一類表達
2. **歸類**：對照現有 36 種模式，分三檔：已覆蓋（無動作）、變體（補識別信號或範例句）、候選新模式
3. **觀察**：候選新模式先記錄，觀察 2–3 輪確認反覆出現才入庫，不追打單點流行語
4. **雙向補樣本**：入庫時 SF/SNF 成對補，明確寫出放行邊界
5. **記錄**：CHANGELOG 記錄觀察來源與判斷過程，不只寫「新增規則」

## 不接受的貢獻

- 純個人偏好的詞彙增刪（需要多來源觀察或誤殺證據）
- 把 skill 改成「模仿某個人風格」的功能（去 AI 味與個人風格是兩件事，本專案只做前者）
- 綁定特定平台的深度改動（本專案保持平台無關）
- 未脫敏的真實文本、未授權的轉錄
- 非 MIT 相容的內容
