# 怎麼跑評測

評測對象是 [benchmark.md](benchmark.md) 的 36 條用例（24 SF ＋ 12 SNF）。目標：

- **SF 通過**：主要問題被改掉，且保護清單無一漂移
- **SNF 通過**：文本被放行，或只收到極輕提示（不動原文）

## 判分標準

每條用例三個判定，全過才算綠：

1. **命中**（SF）／**放行**（SNF）：SF 的「問題點」是否被處理；SNF 是否未被改寫
2. **保真**：價格、數字、真名、連結、引號原話、承諾條款逐項核對，一字不漂
3. **不換湯**：SF 改寫後不得換成同族的另一句空話（例：刪掉「賦能」卻補上「加值」；刪掉「標誌著」卻補上「象徵著」）

長文用例（SF-23、SNF-12）加判第 4 項：

4. **長度節奏**：不擅自刪句縮水；整句空話進「建議刪除（待確認）」清單而不是直接消失；節奏句、數據句原樣保留

## 快速自查（單模型）

改規則後的最小回歸。開一個乾淨 session（不要帶入本 repo 以外的寫作規則）：

```
讀取 SKILL.md 與 references/ 全部檔案，然後逐條處理 evals/benchmark.md 的用例：
- SF 用例：輸出改寫結果
- SNF 用例：輸出「放行」或說明為何不改
輸出格式：用例編號｜判定（改寫/放行）｜改寫結果或理由
```

人工對照「預期方向／預期行為」逐條打 ✅ / ❌。

## 雙模型交叉（發版前）

避免單一模型的口癖影響判分，發版前用兩個不同家族的模型交叉：

1. **改寫端**（例如 Codex）：載入 skill，跑完全部用例，存成 `results-v<版本>.md` 草稿
2. **判分端**（例如 Claude）：不載入 skill，只給它 benchmark 的「原文＋問題點＋預期方向」與改寫端輸出，逐條判 ✅ / ❌ 並附一句理由
3. 有爭議的用例人工終判

### Codex 改寫端指令範例

```bash
cd speak-human-tw
codex exec -C . "讀取 ./SKILL.md 與 ./references/ 下所有檔案，依規則逐條處理 ./evals/benchmark.md 的 36 條用例。SF 輸出改寫結果，SNF 輸出放行判定與一句理由。"
```

### 判分端 prompt 要點

- 明確告知：不換湯規則（換成同族空話記 ❌）
- 明確告知：SNF 被改寫一律記 ❌，即使改寫結果「看起來更好」
- 要求輸出表格：`用例｜判定｜理由（一句）`

## 結果歸檔

每次完整評測存 `evals/results-v<版本>.md`，記錄：

- 日期、改寫端與判分端模型
- SF 通過率、SNF 誤殺率（目標：SF ≥ 90%、SNF 誤殺 0）
- 非綠用例逐條點評與處置（改規則／改用例／接受）

## 新增用例的規範

- SF 與 SNF 盡量成對：新增「該改」規則時，同時想一條「不可誤殺」的邊界
- 用例必須是合成文本或已脫敏，不得指向真實人物、品牌、課程
- 每條固定欄位：原文、問題點（SF）或不能動的理由（SNF）、預期方向／預期行為
- 更新本檔與 benchmark.md 開頭的計數，並在 CHANGELOG 記一筆
